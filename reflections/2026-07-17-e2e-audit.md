# Reflection: End-to-end audit and package hardening

## Task

- **ID / title:** SnapAction end-to-end audit and PR gate
- **Date:** 2026-07-17
- **Scope:** Current SwiftPM app, generated `.app`, safe native interaction surface, local integrations, persistence, telemetry, and PR readiness
- **Authority boundary:** Local changes and safe reversible UI interactions were allowed; Calendar/Reminder writes, TCC changes, system-setting mutations, and release credentials were not

## Success and Risk

- **Success criteria:** Build and tests pass; the generated app launches and behaves correctly; official privacy/access contracts match implementation; local data and logs do not leak payloads; all remaining gaps are explicit
- **Hypothesis 1:** Existing tests covered core workflow logic but not generated-bundle privacy/signing behavior
- **Hypothesis 2:** Runtime inspection would reveal state or persistence gaps not visible in unit tests
- **Hypothesis 3:** Most launch error logs would be framework/service noise rather than SnapAction-owned failures
- **Rollback path:** Revert the focused audit commit; local persisted JSON remains backward-compatible and permission tightening does not change its schema

## Candidate Directions

| Candidate | Expected benefit | Main risk | Evidence before choice | Decision |
|---|---|---|---|---|
| Patch only failing tests | Small diff | Would miss package, permission, runtime, and log defects | Baseline tests were green while strict bundle verification and purpose-string checks failed | Rejected |
| Harden the smallest proven cross-layer defects and verify the real app | Fixes observable contract failures without redesigning the product | More review surface across script, platform adapter, core persistence, and UI | Each retained change had a failing check, runtime signal, or official-document mismatch | Chosen |
| Introduce an Xcode/App Store release pipeline now | Could address all distribution gates | Material scope expansion requiring signing identity, entitlements, assets, and owner decisions | User requested correctness and merge, but no release credentials or product metadata were supplied | Deferred |

## Evidence

- **First meaningful failure signal:** The baseline `.app` lacked required purpose strings and failed strict code-sign verification because its completed Info.plist was not covered
- **Commands or runtime checks:** SwiftPM tests/build, release build, Thread Sanitizer, purpose-string/signature contract, unified logs, live Computer Use scenarios, file modes, and clipboard hash equality
- **What the evidence ruled in or out:** Core workflow remained stable; real defects were concentrated at packaging, least-privilege EventKit access, operation presentation, persistence confidentiality, silent clipboard failures, and log privacy. AppIntents/CoreSpotlight launch errors were isolated to Apple subsystems and did not affect runtime flow.

## Decision

- **Root cause or remaining unknown:** The repository's core logic had stronger test coverage than its generated bundle and platform boundaries. External Screen Recording/EventKit behavior remains unproven without owner-approved permissions/writes.
- **Retained fix / direction:** Contract-test the generated bundle; request write-only Calendar access; secure owned JSON files; expose pasteboard failure; fail closed into a visible startup state when durable storage cannot initialize; centralize busy availability; keep telemetry metadata-only.
- **Why alternatives were rejected:** A formatter migration and full release-pipeline build would be unrelated expansion. Live permission changes and writes exceeded the safe authority boundary.
- **Residual risk:** Apple framework launch noise, unverified permissioned integrations, and non-distributable ad-hoc signing
- **Rollback trigger:** Any regression in launch, persistence migration, pasteboard behavior, EventKit authorization, or core workflow tests

## Reusable Lesson

- **Pattern to retain:** Treat generated bundles, file modes, OS permission APIs, and unified logs as executable contracts alongside unit tests
- **Pattern to avoid:** Calling a green SwiftPM suite an end-to-end pass when package metadata and the real app have not been exercised
- **Where it applies next:** Other local-first macOS utilities that stage `.app` bundles outside Xcode
